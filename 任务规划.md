# 任务规划：Auto-SWE-Deer 落地路线图 (Route B)

本项目采用**应用工程**模式：在DeerFlow 2.0框架之上构建定制化软件工程智能体应用。

---

## 核心原则

- **使用框架，非重建框架**：所有编排层（agent_loop、MessageBus、Sub-agent调度）全部使用DeerFlow原生能力
- **定制边缘，非核心**：专注Sandbox安全、MCP工具链、Skill配置三个差异化领域
- **s_full.py是参考文档**：仅供理解DeerFlow设计理念，不移植其代码
- **⚠️ 三大陷阱防范**：Root Permission Trap、State Handoff、DeerFlow桥接MCP

---

## Phase 1: DeerFlow运行时搭建 + Sandbox定制基座

**核心任务**：`pip install deerflow` + 自定义SandboxController

### Task 1.1: DeerFlow环境初始化
- [ ] 创建Python 3.10+虚拟环境
- [ ] 执行 `pip install deer-flow` 安装DeerFlow 2.0
- [ ] 阅读DeerFlow官方文档，理解其Skill格式和Sub-agent机制
- [ ] 验证DeerFlow CLI能正常启动

### Task 1.2: DeerFlow项目初始化 + MCP桥接配置
- [ ] 执行 `deer-flow init` 创建项目脚手架
- [ ] 研究生成的 `config.yaml` 结构
- [ ] **⚠️ 关键配置**：在 `config.yaml` 中注册MCP服务器（The Bridge陷阱）

```yaml
# config.yaml
mcp_servers:
  sandbox_tools:
    command: "python"
    args: ["mcp_tools/main.py"]
    env:
      DEERFLOW_MODE: "production"
```

- [ ] 研究 `skills/` 目录结构（DeerFlow原生Skill格式）

### Task 1.3: SandboxController开发 [核心定制IP]
- [ ] 创建 `sandbox/` 目录
- [ ] 实现 `SandboxController` 类：
  - `create_container()`: Python Docker SDK拉起容器
  - **`⚠️ 关键`**：传入 `user=f"{os.getuid()}:{os.getgid()}"` 确保权限一致（Root Permission Trap）
  - `mount_worktree()`: 将worktree目录挂载到容器
  - `execute_in_sandbox()`: 在容器内执行命令
  - `destroy_container()`: 销毁容器
- [ ] 实现 `CommandInterceptor`：
  - 危险命令黑名单（`rm -rf /`, `sudo`, `shutdown`等）
  - 路径安全校验（防止路径穿越）
- [ ] 创建 `sandbox/worktree.py`：
  - 封装s12 WorktreeManager核心逻辑
  - 实现Task与Worktree绑定
- [ ] **⚠️ 关键**：创建 `sandbox/registry.py`：
  - 实现 `TaskRegistry` 类
  - 维护 `task_id → SandboxController` 实例的映射
  - MCP Server依赖此模块进行无状态路由

### Task 1.4: 基座验证测试
- [ ] 编写 `test_sandbox_basic.py`
- [ ] 验证：宿主机 → Docker容器通信正常
- [ ] 验证：`rm -rf /` 被CommandInterceptor拦截
- [ ] 验证：Worktree目录正确挂载到容器
- [ ] **⚠️ 关键**：验证容器内创建的文件在宿主机上可正常访问（权限一致性）

## Phase 2: MCP工具服务器开发

**核心任务**：将Sandbox能力封装为MCP协议工具，暴露给DeerFlow调用

### Task 2.1: MCP协议基础
- [ ] 安装MCP SDK：`pip install mcp`
- [ ] 研究MCP stdio服务器标准
- [ ] 创建 `mcp_tools/` 目录结构

### Task 2.2: TaskRegistry集成 [关键]
- [ ] 创建 `mcp_tools/registry_client.py`：
  - 连接 `sandbox/registry.py` 维护的TaskRegistry
  - 提供根据 `task_id` 查询 SandboxController 实例的能力
- [ ] 设计MCP请求协议：每个请求必须携带 `task_id`

### Task 2.3: 文件操作工具 (MCP)
- [ ] 实现 `file_read` MCP工具：
  - **必填参数**：`task_id`, path
  - 路由至对应Sandbox容器内读取
  - 返回文件内容（支持limit行数限制）
- [ ] 实现 `file_write` MCP工具：
  - **必填参数**：`task_id`, path, content
  - 根据 `task_id` 定位到对应的Sandbox + Worktree
  - 强制路由至Sandbox容器内写入
  - 禁止绕过沙盒直接写宿主机

### Task 2.4: Bash执行工具 (MCP)
- [ ] 实现 `bash_execute` MCP工具：
  - **必填参数**：`task_id`, command
  - 根据 `task_id` 路由至对应Sandbox容器
  - 捕获stdout/stderr返回
  - CommandInterceptor前置校验

### Task 2.5: Worktree工具 (MCP)
- [ ] 实现 `worktree_create` MCP工具：
  - **必填参数**：`task_id`, name
  - 调用sandbox/worktree.py创建worktree
  - 与Task ID绑定
- [ ] 实现 `worktree_list` MCP工具
- [ ] 实现 `worktree_remove` MCP工具

### Task 2.6: MCP服务器入口
- [ ] 创建 `mcp_tools/main.py`：
  - 实现MCP stdio协议入口
  - 初始化TaskRegistry连接
  - 注册所有tools的handler
- [ ] **⚠️ 验证**：DeerFlow通过config.yaml能发现并调用此服务器

### Task 2.7: MCP服务器集成测试
- [ ] 启动MCP stdio服务器
- [ ] 使用 `mcp-cli` 或直接通过DeerFlow测试工具调用
- [ ] 验证：`task_id=123` 的file_write不会污染 `task_id=456` 的Worktree（State Handoff隔离）

## Phase 3: SWE Skills配置 (核心定制IP)

**核心任务**：通过DeerFlow原生Skill格式定义Researcher/Coder/Tester行为约束

### Task 3.1: DeerFlow Skill格式研究
- [ ] 阅读DeerFlow官方Skill格式规范（YAML元数据 + Markdown描述）
- [ ] 分析DeerFlow生成的示例Skill结构
- [ ] 理解Skill与Sub-agent的绑定机制

### Task 3.2: Researcher Skill开发 [核心]
- [ ] 创建 `deerflow_app/skills/researcher/` 目录
- [ ] 创建 `skill.yaml`（DeerFlow Skill元数据）：
  - name: researcher
  - description: 代码库研究员
  - tools: [file_read, grep_search]（禁止file_write, bash_execute）
  - constraints: 只读模式
- [ ] 创建 `skill.md`（Markdown行为约束）：
  - 生成分析报告前必须调用grep全文检索
  - 输出结构化诊断书格式
  - 禁止执行任何写操作

### Task 3.3: Coder Skill开发 [核心]
- [ ] 创建 `deerflow_app/skills/coder/` 目录
- [ ] 创建 `skill.yaml`：
  - name: coder
  - description: 代码修改专家
  - tools: [file_read, file_write, bash_execute]
  - constraints: 精准路径引用
- [ ] 创建 `skill.md`：
  - **⚠️ 关键**：修改文件时必须通过 `task_id` 确保操作正确的Worktree
  - 修改文件前必须先完整阅读相关代码
  - 修改后必须调用测试工具验证
  - 测试失败进入反思循环：分析报错 → 精准修改 → 重新测试
  - 最大反思迭代次数N（可配置，默认3）

### Task 3.4: Tester Skill开发 [核心]
- [ ] 创建 `deerflow_app/skills/tester/` 目录
- [ ] 创建 `skill.yaml`：
  - name: tester
  - description: 测试验证专家
  - tools: [bash_execute]
- [ ] 创建 `skill.md`：
  - 主导pytest执行（携带 `task_id` 定位正确的Sandbox）
  - 失败日志提取与格式化
  - 回归验证确认

### Task 3.5: SuperAgent Skill开发
- [ ] 创建 `deerflow_app/skills/superagent/` 目录
- [ ] 创建 `skill.yaml`：
  - name: superagent
  - description: 任务协调主脑
  - capabilities: [意图识别, 任务拆解, Sub-agent孵化]
- [ ] 创建 `skill.md`：
  - 接收用户Issue描述
  - 拆解为Researcher → Coder → Tester工作流
  - **⚠️ 关键**：孵化Sub-agent时确保 `task_id` 被正确传递到后续节点
  - 动态spawn对应Skill的Sub-agent

### Task 3.6: Skill集成验证
- [ ] 在DeerFlow中注册所有Skills
- [ ] 编写测试：发送简单Issue，验证Sub-agent按预期被孵化
- [ ] 验证Researcher确实无法调用Write工具
- [ ] **⚠️ 关键**：验证Coder在修改文件时操作的是正确的Worktree（通过 `task_id` 隔离）

## 关键陷阱检查表

| 陷阱名称 | 检查项 | 验证方法 |
|---------|--------|---------|
| **Root Permission Trap** | `create_container()` 包含 `user=uid:gid` | 容器内创建文件，宿主机可删改 |
| **State Handoff** | 所有MCP工具必填 `task_id` | 并行两个task，检查Worktree隔离 |
| **DeerFlow桥接MCP** | `config.yaml` 包含 `mcp_servers` | DeerFlow能发现并调用MCP工具 |

## 关键认知提醒

> **s_full.py的真正用途**：这是DeerFlow的内部参考实现，用于理解框架设计理念。我们的代码不应该依赖或移植s_full.py的任何模块。所有编排能力必须通过DeerFlow原生API调用。

> **三大陷阱必须防范**：这三个问题在实际部署时会导致严重故障，必须在代码层面强制执行。
