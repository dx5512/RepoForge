# 项目方案：Auto-SWE-Deer (DeerFlow 2.0 应用工程)

## 1. 项目愿景与定位

Auto-SWE-Deer 是基于 ByteDance **DeerFlow 2.0** 框架构建的**软件工程智能体应用**。
系统通过在DeerFlow原生能力之上叠加**自定义沙盒隔离层**和**领域技能卡**，实现从"接收GitHub Issue → 自动克隆 → 沙盒内编译调试 → 补丁生成"的端到端无人值守修复能力。

**核心定位**：不是在DeerFlow内部重写编排引擎，而是利用DeerFlow的原生编排能力，专注于沙盒安全、工具链MCP化、领域技能配置三个差异化方向。

## 3. 核心系统架构

### 3.1 物理执行层 (The AIO Sandbox) - 定制IP

| 组件 | 职责 | 技术实现 |
|------|------|----------|
| SandboxController | 容器生命周期管理 | Python Docker SDK + Git Worktree |
| WorktreeBinder | Task与Worktree绑定 | s12 WorktreeManager核心逻辑 |
| CommandInterceptor | 危险命令拦截 | 黑名单 + 路径安全校验 |

**安全设计**：

- 所有代码修改操作必须路由至Sandbox内的Docker容器
- Worktree提供独立代码副本物理隔离
- 容器销毁后修改自动丢弃，除非显式keep

**⚠️ 关键陷阱：Root Permission Trap**

> 在使用 Python Docker SDK 把宿主机的 `.worktrees/task_xxx` 挂载到容器内时，如果容器默认以 **root** 用户运行，Agent 修改或新建的代码文件权限也会变成 **root**。这会导致宿主机后续提取 `git diff` 或清理目录时报 **Permission Denied** 错误。

**解决方案**：在 `SandboxController.create_container()` 中，必须强制传入宿主机当前用户的 UID 和 GID：

```python
# 正确的做法：确保容器内外的文件读写权限一致
container = client.containers.run(
    image,
    user=f"{os.getuid()}:{os.getgid()}",  # 关键参数！
    volumes={host_path: {"bind": container_path, "mode": "rw"}},
    ...
)
```

### 3.2 工具链层 (MCP Tools) - 定制IP

**⚠️ 关键陷阱：Sub-agent状态传递 (State Handoff)**

> Researcher 查完代码后，主脑唤醒 Coder 继续干活。DeerFlow 原生会传递对话上下文，但 MCP 工具怎么知道当前是哪个任务？如果 Coder 调用 `file_write`，系统必须保证它写的是**同一个 Worktree 目录**，而不是新建一个。

**解决方案**：所有 MCP 工具的入参都必须显式携带 `task_id` 参数。MCP Server 是无状态的，但执行 `file/bash` 操作时，必须依赖传入的 `task_id` 来定位具体的 `SandboxController` 实例和对应的 Worktree 路径。

| 工具 | MCP协议 | 必填参数 | 路由目标 | 安全约束 |
|------|---------|---------|---------|----------|
| file_read | stdio | **task_id**, path | Sandbox容器内 | 路径白名单校验 |
| file_write | stdio | **task_id**, path, content | Sandbox容器内 | 必须挂载Worktree |
| bash_execute | stdio | **task_id**, command | Sandbox容器内 | 危险命令黑名单 |
| worktree_create | stdio | **task_id**, name | 宿主机git | Task绑定校验 |

**MCP Server 无状态设计原则**：

```
MCP Server (无状态)
    │
    ├── 接收 task_id + operation
    │
    ├── 查询 TaskRegistry 获取 SandboxController 实例
    │
    └── 根据 task_id 定位对应的 Worktree 路径执行操作
```

### 3.3 领域技能层 (SWE Skills) - 定制IP

通过DeerFlow原生Skill格式（YAML配置 + Markdown描述）定义：

| 技能卡 | 节点类型 | 核心约束 |
|--------|----------|----------|
| researcher.yaml + researcher.md | DeerFlow Sub-agent | 只读不写，强制Grep前置 |
| coder.yaml + coder.md | DeerFlow Sub-agent | 精准路径+task_id，测试反思循环 |
| tester.yaml + tester.md | DeerFlow Sub-agent | pytest主导，失败日志提取 |

### 3.4 编排层 (DeerFlow Native)

| 能力 | 实现方式 |
|------|----------|
| SuperAgent主控 | DeerFlow原生agent配置 |
| Sub-agent动态孵化 | DeerFlow spawn机制 |
| 长时上下文管理 | DeerFlow Context Engineering |
| 消息路由 | DeerFlow MessageBus原生 |
| 异步后台任务 | DeerFlow Background Tasks原生 |

**⚠️ 关键陷阱：DeerFlow桥接MCP配置 (The Bridge)**

> 虽然我们开发了 MCP Server，但需要让 DeerFlow 发现并调用这个 Server。

**解决方案**：在 DeerFlow 的 `config.yaml` 中注册 MCP 服务器：

```yaml
# config.yaml
mcp_servers:
  sandbox_tools:
    command: "python"
    args: ["mcp_tools/main.py"]
    env:
      # 可选：传递环境变量
      DEERFLOW_MODE: "production"
```

```
        │                      │
        │                      ▼
        │              ┌───────────────┐
        │              │ Tester        │

        │                      │
        └──────────┬───────────┘
                   ▼
┌─────────────────────────────────────────┐
│     SandboxController (我们的IP)         │
│  - Docker容器内执行所有命令              │
│  - Git Worktree隔离                     │
│  - 危险命令拦截                         │

└─────────────────────────────────────────┘
                   │
                   ▼
           git diff → 报告 → IM推送
```

**task_id 隐式传递机制**：

```
DeerFlow SuperAgent
    │
    ├── 创建 Task #123
    ├── 绑定 Worktree "wt/task_123"
    └── 孵化 Coder Sub-agent
            │
            └── DeerFlow 自动在上下文注入：
                [Current Task] task_id=123, worktree=wt/task_123
                        │
                        ▼
                Coder 调用 MCP file_write
                        │
                        └── MCP Server 根据 task_id=123
                            定位到对应的 Sandbox + Worktree
```

## 7. 与原方案的核心差异

| 维度 | 原方案（错误） | 新方案（Route B） |
|------|---------------|------------------|
| 架构认知 | 移植s_full.py重写框架 | 在DeerFlow上构建应用 |
| 编排层 | 手动实现MessageBus等 | 100%使用DeerFlow原生 |
| Phase重心 | 大量底层编码 | 配置+集成+定制边缘 |
| 工程价值 | 造轮子 | 集成创新+安全定制 |
| 权限处理 | 忽略 | ⚠️ user=uid:gid显式处理 |
| 状态传递 | 无方案 | ⚠️ task_id全链路传递 |
| MCP集成 | 只写代码 | ⚠️ config.yaml显式注册 |

## 8. 三大关键陷阱总结

| 陷阱名称 | 位置 | 解决方案 | 关键代码 |
|---------|------|---------|---------|
| Root Permission Trap | SandboxController.create_container() | 传入 `user=f"{os.getuid()}:{os.getgid()}"` | ⚠️ 必须项 |
| State Handoff | MCP工具调用链 | 所有工具必填 `task_id` 参数 | ⚠️ 路由依据 |
| DeerFlow桥接MCP | config.yaml | 注册 `mcp_servers.sandbox_tools` | ⚠️ 发现机制 |
